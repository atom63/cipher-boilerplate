# Cipher Brand and Messaging Foundation

## Overview

This document defines the **brand, narrative, and messaging foundation** for **Cipher**, a Figma plugin focused on **importing, exporting, and synchronizing design tokens**.

Its purpose is to guide how Cipher is described across:

- plugin marketplace copy
- website copy
- feature descriptions
- launch messaging
- marketing assets
- product UI language
- documentation and supporting content

It also provides strategic direction for the broader identity system, including the logo, visual language, and product presentation. But its primary role is not to act as a formal corporate brand strategy. Instead, it is a **practical source of truth for product positioning, messaging, and content drafting**.

Cipher is a self-initiated project with a single creative owner. It does not need to sound generic, neutral, or universal. In fact, part of its strength comes from having a clear point of view.

**Cipher is a highly opinionated token system for Figma.**

That idea should not be treated as a disclaimer. It is a core part of the product’s value and a defining part of how the brand should communicate.

## 2. Brand Premise

Design systems depend on structure beneath the surface.

Color, spacing, typography, radius, motion, and semantic decisions all rely on a hidden layer of logic that keeps products coherent across files, tools, and teams. But that logic is often fragmented, hard to read, and difficult to maintain.

**Cipher exists to make that hidden system logic readable, portable, and synchronized.**

It is not a decorative add-on or a branding gimmick. It is infrastructure: a precise, dependable layer that helps design systems stay aligned.

Cipher is built on a simple belief:

**clarity comes from structure**

Not every tool needs to be infinitely flexible to be valuable. In many cases, a strong and well-defined opinion creates a better workflow than endless configurability. Cipher is built around that philosophy.

## 3. Product Definition

**Cipher** is a Figma plugin for **importing, exporting, and synchronizing design tokens**.

At a functional level, it helps teams:

- bring token data into Figma
- push token data out to other systems
- manage variables and styles more efficiently
- reduce fragmentation across token sources
- maintain consistency between design and code
- support a more reliable source of truth

At a product philosophy level, Cipher is not trying to be a universal tool for every team and every workflow.

It is:

- structured
- deliberate
- workflow-driven
- system-first
- opinionated by design

The product should be described as:

**a highly opinionated token system for Figma**

That means Cipher is built around a specific way of organizing token logic, managing system structure, and keeping design decisions aligned. It is not about offering every possible path. It is about offering a clearer one.

## 4. Product Messaging Objective

Cipher’s messaging should help users quickly understand three things:

### 1. What it is

A Figma plugin for importing, exporting, and synchronizing design tokens.

### 2. What kind of product it is

A highly opinionated token system built around structure, readability, and consistency.

### 3. Why that matters

Because token workflows become fragmented quickly, and a clear system reduces maintenance, ambiguity, and manual overhead.

This means Cipher should not only describe what it does. It should also describe the value of its point of view.

The messaging should consistently communicate that Cipher is for people who value:

- system clarity
- reliable workflows
- readable structure
- better alignment between design and code
- less manual maintenance
- stronger token discipline

### Terminology standards (apply everywhere)

Use these terms consistently across website, plugin listing, and docs:

- **token** = design token data (import/export context)
- **variable** = Figma variable (in-product management context)
- **collection** = Figma variable collection
- **mode** = value mode within a collection (for example, Light and Dark)
- **synchronize** = preferred long form in strategy copy
- **sync** = acceptable short form in UI labels and tight marketing surfaces

## 7. Problem and Opportunity

### The Problem

Design systems often break down not because the visuals are weak, but because the underlying logic becomes fragmented.

Token values may exist in multiple files, vary across tools, drift from code, lose structure over time, or become difficult to manage consistently. Designers and developers may both be working with the same system in theory, but not in the same structure in practice.

Many tools respond to this by becoming more flexible, more configurable, and more complex. But more flexibility does not always create more clarity.

### The Opportunity

There is value in a tool that makes the hidden layer of a design system easier to read and easier to keep aligned.

Cipher turns token management into something more legible and more structured. It reduces friction, increases trust, and gives the system a stronger sense of continuity.

Its opportunity is not just to help teams sync tokens, but to do so through a workflow philosophy that is more deliberate, more readable, and more dependable than generic tooling.

## 11. Core Narrative

Most digital products rely on systems people never see.

Behind every interface is a deeper structure: the token values, naming rules, semantic layers, and implementation logic that keep the experience coherent across screens and surfaces. When that structure is fragmented, the system becomes harder to scale, harder to maintain, and harder to trust.

Cipher is built around the idea that this invisible layer matters.

It helps teams decode, translate, and synchronize design token structure so the system beneath the interface remains readable and aligned. It supports the work quietly by making complexity more usable.

Cipher also assumes that clarity comes from structure, not from limitless freedom. It is a tool with opinions — designed to help users move faster and more confidently within a well-defined system logic.

## 18. Feature Framing Principles

When writing product features, Cipher should not sound like a list of disconnected utilities. Features should be framed as part of a structured workflow.

### Good framing patterns

* Emphasize clarity, structure, and maintenance
* Show how a feature reduces fragmentation
* Connect the feature to system alignment
* Explain why the workflow is intentionally designed that way
* Present batch actions and sync operations as efficiency through structure

### Example feature framing directions

Instead of:

* Rename variables in bulk

Prefer:

* Batch rename variables with more structure and less manual cleanup

Instead of:

* Export tokens to JSON

Prefer:

* Export token logic into structured formats that stay easier to maintain across tools

Instead of:

* Find broken variable links

Prefer:

* Detect broken variable relationships and restore alignment faster

Instead of:

* Create styles automatically

Prefer:

* Generate structured variables and styles from a clearer system foundation
